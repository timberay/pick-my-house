# i18n (Korean / English) Support — Design

**Date**: 2026-04-25
**Status**: Approved (ready for implementation plan)
**Scope**: Add equal-tier Korean / English support to the entire Pick My House app — UI strings, model/error messages, checklist domain data, and date/time formatting. Treat both locales as first-class. No user authentication exists today, so locale preference is stored in a cookie only.

---

## 1. Goals & Non-Goals

### Goals
- Korean and English receive equal treatment (translation quality, UX, formatting).
- URL-based locale routing (`/ko/...`, `/en/...`) — preserves locale across share/copy.
- Discoverable locale switcher in the header.
- First-time visitor experience respects `Accept-Language`.
- Returning visitor experience respects last-chosen locale (cookie).
- Translation parity is enforced in CI (no missing keys ship to production).

### Non-Goals
- Translating user-generated content (house alias, address, agent contact, memo). The user types these in their own language; the app preserves them as-is.
- A user-account model storing locale preference. None exists yet; cookie is sufficient. Revisit if/when auth is added.
- Locales beyond `:ko` and `:en`. Architecture should not block adding `:ja` later, but no work for it is in scope now.
- Right-to-left layouts. Not needed for `:ko` / `:en`.
- Translation Management System (TMS) integration. Translators edit YAML directly.

## 2. Architecture Overview

```
HTTP request: /en/houses (or /houses, or /)
        │
        ▼
Routes — scope "(:locale)", locale: /ko|en/
        │
        ▼
ApplicationController — Localized concern (around_action)
   1. params[:locale]            (URL prefix)
   2. cookies[:locale]           (last choice)
   3. Accept-Language header     (browser pref)
   4. I18n.default_locale (=:ko) (fallback)
        │
        ▼
I18n.with_locale(locale) { render }
        │
        ▼
Translation sources:
  • config/locales/ko.yml, en.yml  ← all translatable text
  • config/checklist.yml           ← structure (keys) only, no labels
```

**Key principles**

- Single source of truth for translatable text = `config/locales/{ko,en}.yml`.
- `config/checklist.yml` manages **structure only**: which domains exist, which item keys belong to each. All labels live in locale YAML under the `checklist.*` namespace.
- Routes use **optional** `(:locale)` scope, so `/houses` continues to work via the fallback chain — bookmarks don't break.
- `default_url_options` injects the current `:locale` into every link helper, so navigating within the app preserves the locale.
- Unsupported locales (`/fr/...`) are rejected at the routing layer (`RoutingError`, 404) — no silent redirect.

**Routing decisions**

- `/` redirects (302) to `/<resolved-locale>` so the URL is always canonical at the root.
- `/houses` (no locale prefix) does **not** redirect to `/ko/houses` — bookmarks and externally-shared links continue to render. The fallback chain still applies for translation lookup.
- Unknown locale segment → 404, not a graceful fallback. This avoids confusing canonicalization issues.

## 3. Components

Each unit has a single responsibility and a clear interface.

### 3.1 `LocaleResolver` (PORO — `app/lib/locale_resolver.rb`)

**Responsibility**: Pure function that picks a locale from inputs.

```ruby
LocaleResolver.call(
  param: nil_or_string,
  cookie: nil_or_string,
  accept_language: nil_or_string
) # => :ko or :en
```

- Returns the first input that resolves to a member of `I18n.available_locales`.
- For `accept_language`, uses the `http_accept_language` gem (already a Rails standard) to do quality-aware parsing.
- No side effects. Trivial to unit-test.

**Why a PORO and not a controller method**: keeping the fallback chain pure makes it easy to test exhaustively (the table of cases is the spec) and to call from non-controller contexts (e.g., the root redirect lambda in `routes.rb`).

### 3.2 `Localized` concern (`app/controllers/concerns/localized.rb`)

**Responsibility**: Per-request locale resolution and persistence.

```ruby
module Localized
  extend ActiveSupport::Concern

  included do
    around_action :switch_locale
  end

  def default_url_options
    { locale: I18n.locale }
  end

  private

  def switch_locale(&action)
    locale = LocaleResolver.call(
      param: params[:locale],
      cookie: cookies[:locale],
      accept_language: request.headers["Accept-Language"]
    )
    if locale.to_s != cookies[:locale]
      cookies[:locale] = { value: locale.to_s, expires: 1.year.from_now }
    end
    I18n.with_locale(locale, &action)
  end
end
```

Included in `ApplicationController`. Controller actions don't change.

### 3.3 Locale switcher partial (`app/views/shared/_locale_switcher.html.erb`)

**Responsibility**: Render the header toggle (`한국어 / English`).

- Implemented as an ERB partial to match the existing pattern (`_back_link.html.erb`, `_severity_badge.html.erb` etc. — the project does not use ViewComponent).
- Active locale gets distinct styling (e.g., underline / weight).
- Inactive locale links to the same path with `:locale` swapped via `url_for(locale: :en)` (or `:ko`).
- The cookie update happens automatically on the next request via `Localized#switch_locale` — the partial itself doesn't write cookies.
- Rendered from `app/views/layouts/_header.html.erb` aligned to the right side. Inline labels separated by `·` so it fits the mobile header without an extra row.

### 3.4 `Checklist` module changes (`app/lib/checklist.rb`)

**Before**: `Domain` and `Item` structs hold `label_ko`. Labels live in `config/checklist.yml`.

**After**: structs hold structure only.

```ruby
Domain = Struct.new(:key, :items, keyword_init: true)
Item   = Struct.new(:key, :domain, keyword_init: true)
```

Callers obtain labels via:

```ruby
I18n.t("checklist.domains.#{domain.key}.label")
I18n.t("checklist.items.#{item.key}.label")
```

### 3.5 `ChecklistHelper` (new — `app/helpers/checklist_helper.rb`)

**Responsibility**: Convenience wrappers so views don't repeat the `I18n.t` key construction.

```ruby
module ChecklistHelper
  def domain_label(domain) = t("checklist.domains.#{domain.key}.label")
  def item_label(item)     = t("checklist.items.#{item.key}.label")
end
```

### 3.6 Root redirect (`config/routes.rb`)

**Responsibility**: Send `/` to `/<resolved-locale>` so the root URL is canonical.

The current root route is `root "houses#index"`. Replace it with a redirect lambda:

```ruby
root to: redirect { |_, req|
  locale = LocaleResolver.call(
    param: nil,
    cookie: req.cookies["locale"],
    accept_language: req.env["HTTP_ACCEPT_LANGUAGE"]
  )
  "/#{locale}"
}
```

The redirected URL `/ko` (or `/en`) then matches `houses#index` inside the `(:locale)` scope, preserving current behavior.

### 3.7 Layout updates (`app/views/layouts/`)

- `<html lang="<%= I18n.locale %>">` — accessibility / SEO.
- `<link rel="alternate" hreflang="ko" href="...">` and `hreflang="en"` — search-engine signal that both URLs serve equivalent content.
- `_header.html.erb` renders `LocaleSwitcherComponent`.

## 4. Translation Key Structure

Standard Rails namespaces (`activerecord.*`, `errors.*`, `helpers.*`, `date.*`, `time.*`) are kept as-is. App-specific text is grouped by view path so Rails' lazy lookup (`t(".title")`) resolves automatically.

```yaml
ko:
  # === Rails standard ===
  activerecord:
    models:
      house: "집"
      inspection_check: "점검 항목"
    attributes:
      house: { alias: "집 별칭", address: "주소", agent_contact: "중개인", visited_at: "방문일" }
      inspection_check: { item_key: "항목", severity: "평가", memo: "메모" }
    errors:
      models:
        house:
          attributes:
            alias:
              blank: "집 별칭을 입력해주세요"
              too_long: "집 별칭은 최대 %{count}자까지 가능합니다"
  errors:
    messages:
      blank: "필수 항목입니다"
      taken: "이미 사용 중입니다"
      too_long: "최대 %{count}자까지 가능합니다"
      inclusion: "허용되지 않은 값입니다"
  helpers:
    submit: { create: "추가", update: "수정", submit: "저장" }
  date:
    formats: { default: "%Y-%m-%d", long: "%Y년 %m월 %d일" }
  time:
    formats: { default: "%Y-%m-%d %H:%M" }

  # === App-specific: domain data ===
  checklist:
    domains:
      water:    { label: "수도/배관" }
      electric: { label: "전기" }
      mold:     { label: "곰팡이/결로" }
      windows:  { label: "창호/단열" }
      smell:    { label: "냄새" }
      # ... all domains in config/checklist.yml
    items:
      water_pressure: { label: "수압 충분 (샤워기 세게 틀어서)" }
      hot_water_time: { label: "온수 30초 내 나옴" }
      # ... all items in config/checklist.yml

  inspection_checks:
    severity: { ok: "양호", warn: "주의", severe: "심각" }

  # === App-specific: UI text — grouped by view path for lazy lookup ===
  layout:
    app_name: "Pick My House"
    locale_switcher: { ko: "한국어", en: "English" }
  common:
    back: "뒤로"
    save: "저장"
    cancel: "취소"
    edit: "수정"
    destroy: "삭제"
    confirm_destroy: "정말 삭제하시겠어요?"
  houses:
    index: { title: "내 집 목록", new: "집 추가" }
    show:  { title: "..." }
    new:   { title: "새 집 추가" }
    edit:  { title: "집 수정" }
    form:  { ... }
  inspection_checks:
    check_row: { ... }
  summaries:
    show: { title: "요약" }
  shared:
    back_link: { ... }
    severity_badge: { ... }
    domain_progress: { ... }
```

**Naming rules**

- Keys are `lowercase_snake_case`.
- One view = one namespace (`houses.index.*`).
- Reusable actions live in `common.*`.
- Lazy lookup is the default: `t(".title")` inside `app/views/houses/index.html.erb` resolves to `houses.index.title`.

**`en.yml`** mirrors the same tree with English values. The two files must have identical key sets — enforced by `i18n-tasks`.

### `config/checklist.yml` after the change

Keep structure only (no labels). The simplest form:

```yaml
water:
  items: [water_pressure, hot_water_time, rust_free, drain_speed, balcony_drain, toilet_flush, boiler_on_time]
electric:
  items: [outlets_working, breaker_accessible, lights_working, aircon_drain, internet_line]
# ...
```

`Checklist` loads this and exposes `Domain { key, items: [Item { key, domain }] }`.

## 5. Migration Strategy (Tidy First)

Structural changes and behavioral changes never share a commit.

### Phase A — Structural only (app stays Korean-only)

A1. **Restructure `config/checklist.yml`** + update `Checklist` module
   - YAML: `domain → items: [keys]`.
   - Drop `label_ko` from `Domain` and `Item` structs.
   - Move existing Korean labels into `config/locales/ko.yml` under `checklist.*`.
   - Add `ChecklistHelper#domain_label`, `#item_label`.
   - Update view callers (`_domain_progress.html.erb` etc.) to use the helper.
   - Tests: existing tests pass; the rendered output is unchanged.

A2. **Extract hardcoded Korean strings from views/components**
   - For each of the 12 files containing Korean text, replace literals with `t(".key")`.
   - Add the keys to `config/locales/ko.yml` under the matching view-path namespace.
   - Tests: existing system tests still assert the same Korean text — they keep passing.

At the end of Phase A the app is **still Korean-only**, but the structure supports a second locale cleanly.

### Phase B — Behavioral (add English)

B1. **`LocaleResolver` PORO** + unit tests covering the full fallback table.
B2. **`Localized` concern** + include in `ApplicationController`.
B3. **Routes**: `(:locale)` scope, root redirect, locale constraint.
B4. **`LocaleSwitcherComponent`** + insert into `_header.html.erb`.
B5. **`config/locales/en.yml`** — full English mirror of `ko.yml`.
B6. **Add `i18n-tasks` gem**, add `bundle exec i18n-tasks missing` and `unused` to CI.

### Phase C — Polish

C1. Date/time formats per locale (`date.formats`, `time.formats`).
C2. `<html lang="...">` dynamic in layout.
C3. `<link rel="alternate" hreflang="...">` SEO meta.
C4. System smoke test on the English locale path.

## 6. Error Handling & Edge Cases

| Situation | Behavior |
|-----------|----------|
| `/fr/houses` (unsupported locale) | Routes constraint `locale: /ko\|en/` → 404 `RoutingError`. |
| `params[:locale]` absent | `LocaleResolver` fallback chain (cookie → Accept-Language → default). |
| Missing translation key | dev/test: raise `I18n::MissingTranslation` (caught immediately). prod: fall back to `default_locale` and `Rails.logger.warn`. CI's `i18n-tasks` is the first defense. |
| `Accept-Language: en-US,en;q=0.9,ko;q=0.8` | Parsed via `http_accept_language` gem — quality-aware match against `I18n.available_locales`. |
| User-generated content (alias, memo, address, agent contact) | Not translated. Rendered as the user typed it. |
| Cookies disabled | `Set-Cookie` is best-effort. Subsequent requests fall back to `Accept-Language`. No errors. |
| Bookmarked legacy URL `/houses` (no locale prefix) | Optional scope matches → renders normally with fallback-resolved locale. **No** redirect to canonical — keeps old bookmarks working. |
| Form validation failure | `activerecord.errors` namespace → translated automatically in the active locale. |
| Untrusted cookie value (`'"; DROP TABLE`, etc.) | `LocaleResolver` validates membership in `I18n.available_locales` — anything else is ignored. |

**Key decisions captured**

- Missing-translation in prod: log + fall back. Not silent.
- No canonical redirect for prefix-less URLs: bookmark stability outweighs SEO purity.
- `http_accept_language` gem is a worthwhile dependency over hand-rolled parsing.

## 7. Testing Strategy

### Unit (Minitest, `test/`)

**`test/lib/locale_resolver_test.rb`** — the core. Exhaustive table:
- `param: 'en'` → `:en`.
- `param: nil, cookie: 'ko'` → `:ko`.
- `param: nil, cookie: nil, accept_language: 'en-US,en;q=0.9,ko;q=0.8'` → `:en`.
- All inputs nil → `I18n.default_locale` (`:ko`).
- `param: 'fr'` (unsupported) → ignored, falls through to next source.
- `param: ''` → ignored.
- `cookie: '\"; DROP TABLE'` → ignored.
- Adding `:ja` to `I18n.available_locales` does not require resolver changes.

**`test/lib/checklist_test.rb`** — update existing tests:
- `Checklist.domains` returns domain keys + item lists; no label methods.
- `Checklist.item(key)` returns an `Item` with `:key` and `:domain`.
- Malformed YAML raises `Checklist::Error`.

**`test/helpers/checklist_helper_test.rb`** — new:
- `I18n.with_locale(:ko) { domain_label(d) }` returns the Korean label.
- `I18n.with_locale(:en) { item_label(i) }` returns the English label.
- Missing translation raises in test env (catches gaps in CI).

### Integration (`test/integration/`)

**`test/integration/locale_routing_test.rb`**:
- `GET /ko/houses` → 200, Korean page.
- `GET /en/houses` → 200, English page.
- `GET /fr/houses` → 404.
- `GET /houses` (no cookie, `Accept-Language: en`) → 200, English page.
- `GET /houses` (`cookie: locale=ko`, `Accept-Language: en`) → 200, Korean (cookie wins).
- `GET /` (no cookie) → 302 → `/ko`.
- `GET /` (`Accept-Language: en`) → 302 → `/en`.
- `GET /en/houses` response sets `locale=en` cookie.

### System (`test/system/`)

**`test/system/locale_switcher_test.rb`**:
- Visit `/ko/houses` → header shows `한국어` active, `English` link.
- Click `English` → URL becomes `/en/houses`, page renders in English.
- Revisit later → cookie sends user straight to English without further interaction.
- Submit a form with a validation error → error message renders in the active locale.

**Existing system tests**: keep them on `:ko` (no rewrite needed). Add one English smoke scenario; trust `i18n-tasks` to catch the rest.

### Static (CI)

**`i18n-tasks`**:
- `bundle exec i18n-tasks missing` — must report 0 (key sets in `ko.yml` and `en.yml` are identical).
- `bundle exec i18n-tasks unused` — must report 0 (or items are explicitly ignored in `config/i18n-tasks.yml`).

Both commands run in CI and fail the build on non-zero output.
